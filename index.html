Transformed a manual, error-prone HR evaluation process into a centralized web platform — serving 400 employees across competency reviews, 360° evaluations, and self-assessments.

Overview

🔴 The Problem

  • 400-employee annual evaluation managed entirely in Excel — assignment errors across 40+ departments
  • No real-time visibility into progress; HR followed up manually with every individual
  • Separate tools for competency, 360°, and self-assessment created data silos
  • Post-cycle result compilation required days of manual Excel consolidation

🟢 The Solution

  • One-click auto-assignment engine mapping evaluators to targets by tier (T1–T4)
  • Live dashboard with overall %, department-level, and per-person breakdown
  • Unified portal for all three evaluation types in a single system
  • Structured Excel export for downstream HR analytics — ready in seconds

👤 My Role

Under a PM, I owned the full technical delivery: requirements sessions with the HR team, translating business needs into system design, database schema, frontend for admin and participant portals, API routes, assignment logic, and post-deployment operations support.

Overview
Architecture

🔴 The Problem

During a leadership 360° survey cycle, the ops team needed to send personalized reminders to 150 leaders. Each message required a unique leader name, deadline date, and survey URL:

  • 4 people × 2 hours each = 8 person-hours per campaign
  • Manual copy-paste from Excel for every single recipient
  • Fatigue-driven errors — occasional wrong-name or wrong-link sends
  • No send confirmation or systematic error tracking

🟢 My Discovery Process

I ran a requirements session to understand:

  • Which variables changed per recipient — name, deadline, survey link
  • What approval checkpoints they needed before mass-sending
  • How they wanted to track status and errors
  • Key insight: biggest fear was sending wrong content to 150 people at once → designed explicit human review gate

Key Design Decisions

  • Real-time preview — operator sees rendered message as they type; automation without visibility doesn't build trust
  • Explicit human gate — approval required before dispatch; designed from stated trust concerns, not added after
  • Job log as product feature — logging is accountability infrastructure, not a debugging tool

What I Learned

  • The preview panel was more valuable than the automation itself — it made the output tangible
  • Friction at the right moment (review gate) increased operator confidence significantly
  • For ops teams, "did it actually send?" matters more than any feature — log first

Resume & Job Search
Automation Pipeline

End-to-end pipeline: scrapes job postings with Browserbase, parses PDF resume, runs keyword-level JD fit analysis, and surfaces bullet-by-bullet optimization gaps.

Overview
Architecture
Workflow
Screenshots
Engineering

🔴 The Problem

  • 30–60 min per application to read a JD and map keywords to resume bullets manually
  • Easy to miss high-priority terms buried in lengthy requirements sections
  • No systematic way to compare keyword coverage across multiple applications
  • Manual process doesn't scale across 10+ simultaneous applications

🟢 Solution Design

  • Collection — Browserbase scraper crawls job boards into structured jobs.csv
  • Parsing — PDF resume parser with raw-text fallback for every failure case
  • Analysis — Rule-based NLP extracts JD keywords by section weight and frequency
  • Matching — Bullet-by-bullet coverage flags missing high-priority keywords

Architecture Principles

  • Layered, independently testable — extract → parse → analyze → match each runs in isolation
  • Raw text preserved everywhere — parse failures are always recoverable from the original input
  • Deterministic base — rule-based is debuggable and trustworthy; LLM is a later opt-in layer
  • Edit UI is first-class — 80% automation means 20% lives in a good edit interface

Key Takeaways

  • Highlighting missing job description keywords is more actionable in building own resume
  • PDF parsing inconsistency is a product constraint to accept, not a problem to "solve"
  • Rewrite suggestions without user control are dangerous — always show what changed
  • Job scraping and resume analysis share the same need: clean, structured, comparable text
